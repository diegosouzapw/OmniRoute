## Summary

- Implements `VisionBridgeGuardrail` (priority 5) that intercepts image-bearing requests to non-vision models
- Extracts descriptions via a configurable vision model (default: `gpt-4o-mini`), replaces images with text, and forwards the modified request
- Fails open on any error — never blocks valid requests

## Details

When a user sends an image to a text-only model (e.g., MiniMax, GLM, DeepSeek), OmniRoute currently either returns an error or silently drops the image. Vision Bridge solves this by:

1. **Detecting** image content parts (`image_url` and base64 `image` formats) in requests to non-vision models
2. **Extracting** descriptions by calling a configurable vision model via OpenAI-compatible API
3. **Replacing** image blocks with labeled text descriptions (e.g., `[Image 1]: A sunset over the ocean`)
4. **Forwarding** the modified text-only request to the target model

### Architecture Decisions

- **AD1**: Priority 5 (runs BEFORE PIIMasker) — needs original image data before PII stripping
- **AD2**: Direct `fetch()` call — simpler than full executor pipeline for a single vision call
- **AD3**: Fail-open — Vision Bridge is a convenience feature, never blocks requests
- **AD4**: OpenAI-compatible format — portable across vision providers
- **AD5**: Settings-based config — consistent with OmniRoute's existing settings pattern
