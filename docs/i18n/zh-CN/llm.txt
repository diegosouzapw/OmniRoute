> OmniRoute 是一个免费、开源的 AI 网关，为多个大语言模型服务商提供通用 API 代理。通过单一 OpenAI 兼容端点，提供智能路由、自动容灾、负载均衡和格式转换功能，覆盖 177 个 AI 服务商。内置 MCP Server（37 个工具）、A2A v0.3 协议、记忆/技能系统、云代理（Codex Cloud、Devin、Jules）、安全护栏框架，以及 Electron 桌面应用。

## 概述

OmniRoute 解决了管理多个 AI 服务商订阅、配额和速率限制的问题。它位于你的 AI 驱动工具（IDE 编程助手、CLI 工具）与 AI 服务商之间，通过 4 层容灾体系智能路由请求：订阅 → API Key → 低成本 → 免费。

**核心价值：** 一个端点（`http://localhost:20128/v1`），无限模型，零停机，最低成本。

**当前版本：** 3.8.8

## 技术栈

- **运行时：** Node.js `>=22.22.2 <23 || >=24.0.0 <27`，ES Modules（`"type": "module"`）
- **框架：** Next.js 16（App Router）+ TypeScript 5.9
- **数据库：** SQLite + better-sqlite3（本地、零配置、55 个迁移）
- **状态管理：** Zustand（客户端），SQLite（服务端持久化）
- **UI：** React 19、Tailwind CSS 4、Recharts（分析图表）、@lobehub/icons（130+ 服务商 SVG 图标）
- **认证：** 服务商侧 OAuth 2.0（PKCE），本地用户侧 bcrypt
- **Schema：** Zod v4（所有 API / MCP 输入校验）
- **后台任务：** 自定义 Token 健康检查调度、24 小时模型自动同步
- **流式传输：** Server-Sent Events（SSE）实时代理响应
- **代理引擎：** 自定义管线，包含格式转换、熔断器、速率限制、Auto-Combo 引擎
- **国际化：** next-intl，40+ 种语言
- **桌面：** Electron（跨平台：Windows、macOS、Linux）
- **包发布：** npm（`omniroute`）和 Docker Hub（`diegosouzapw/omniroute`）

## 关键架构决策

1. **OpenAI 兼容 API 接口：** 所有入站请求遵循 OpenAI API 格式。这使 OmniRoute 可以作为任何支持自定义 OpenAI 端点工具的直接替代。

2. **通过格式转换器抽象服务商：** 每个 AI 服务商在 `open-sse/translator/` 中都有一个转换器，透明地在 OpenAI 格式和该服务商原生格式之间转换。

3. **基于连接的服务商模型：** 服务商以"连接"形式存储在 SQLite 中。每个连接有 `id`、`provider`、`authType`（oauth/apikey/free）、`isActive` 标志和凭证。同一服务商可创建多个连接，用于多账户轮换。

4. **Combo 系统实现容灾：** 用户创建"Combo"——即 `服务商/模型` 对的有序列表。代理按顺序尝试每个选项，直到成功。支持 14 种策略，包括带自修复的 Auto-Combo 和用于会话连续性的 Context Relay。

5. **SSE 代理管线：** 代理管线基于中间件：请求 → 认证解析 → 速率限制 → 熔断器 → 格式转换 → 上游调用 → 响应转换 → SSE 流返回客户端。

6. **SQLite 持久化：** 所有状态（服务商、Combo、日志、设置、API Key、记忆、技能）通过 21 个领域专用模块存储在单一 SQLite 数据库中。所有数据库操作均通过 `src/lib/db/` 模块进行，路由中绝不出现裸 SQL。

7. **OAuth + PKCE：** OAuth 流程使用 PKCE 保障安全。Token 刷新由后台任务（`tokenHealthCheck.ts`）处理。

8. **ProviderIcon 组件：** 统一图标系统，使用 `@lobehub/icons`（130+ SVG）+ PNG 降级 + 通用图标降级链。在服务商、仪表盘和代理页面上使用。

9. **数据库架构：** `localDb.ts` 仅为再导出层——真正的逻辑位于 21 个 `src/lib/db/` 模块和 16 个 SQL 迁移中。

10. **上游响应头：** 自定义响应头在执行器中合并，位于默认认证之后；相同响应头名称会替换执行器值。禁用的响应头名称定义在 `src/shared/constants/upstreamHeaders.ts`。

11. **记忆/技能跨系统：** 记忆和技能影响 MCP 工具、请求管线和 A2A 技能。记忆提供跨会话的持久上下文；技能提供带沙箱隔离的可扩展工具执行。

12. **领域策略引擎：** `src/domain/` 包含策略引擎模块（policyEngine、comboResolver、costRules、degradation、fallbackPolicy、lockoutPolicy、modelAvailability、providerExpiration、quotaCache、configAudit），独立于管线管控路由决策。

13. **服务商常量加载时校验：** 所有服务商定义在模块加载时通过 Zod Schema 校验（`src/shared/validation/providerSchema.ts`）。无效的服务商会立刻暴露。

## 主要流程

### 代理请求流程
1. 客户端向 `/v1/chat/completions` 发送 OpenAI 格式请求
2. API Key 校验
3. 模型解析：直接模型查询或 Combo 查询
4. 对于 Combo：按选定策略遍历模型
5. 认证解析：获取目标服务商的凭证
6. 格式转换：OpenAI → 服务商原生格式
7. CLI 指纹匹配（若对该服务商启用）
8. 上游请求，带熔断器和速率限制
9. 响应转换：服务商 → OpenAI 格式
10. omniModel 标签清理（剥离内部标签）
11. SSE 流式返回客户端
12. 记忆提取（若记忆系统启用）
13. 用量日志记录和成本计算

### OAuth 流程
1. 仪表盘发起 `/api/oauth/[provider]/authorize`
2. 用户在浏览器中完成 OAuth 登录
3. 回调命中 `/api/oauth/[provider]/exchange`
4. Token 作为服务商连接存储在 SQLite 中
5. 后台任务在过期前刷新 Token

## LLM 重要注意事项

1. **存在两个模型端点：** `/api/models`（仪表盘，全部模型）和 `/v1/models`（OpenAI 兼容，仅活跃模型）。

2. **服务商 ID 与别名：** 服务商同时有 ID（`claude`、`github`）和短别名（`cc`、`gh`）。模型引用格式为 `alias/model-name`（如 `cc/claude-opus-4-6`）。

3. **`open-sse/` 目录是独立的 npm workspace**，拥有自己的配置、处理器、执行器、转换器和服务。

4. **环境变量：** 所有配置位于 `.env`（来自 `.env.example`）。关键变量：`PORT`、`NEXT_PUBLIC_BASE_URL`、`API_KEY`、`ADMIN_PASSWORD`。

5. **数据库层：** 操作通过 `src/lib/db/` 模块（45+ 个领域专用文件，55 个迁移）进行。`localDb.ts` 仅做再导出——新增函数请加到正确的 `db/*.ts` 模块。

6. **测试** 使用 Node.js 内置测试运行器 + Vitest。运行 `npm test`。MCP/autoCombo 用 Vitest（`npm run test:vitest`）。E2E 用 Playwright（`npm run test:e2e`）。覆盖率门禁：75% statements/lines/functions，70% branches。

7. **MCP 和 A2A 页面是嵌入到 `/dashboard/endpoint` 内的选项卡**，不是独立路由。

8. **ACP 代理** 位于 `src/lib/acp/registry.ts`，有检测缓存。自定义代理通过设置 DB 存储。

9. **Auto-Combo 引擎** 位于 `open-sse/services/autoCombo/` — **9 因子评分**（health 0.22、quota 0.17、costInv 0.17、latencyInv 0.13、taskFit 0.08、specificityMatch 0.08、stability 0.05、tierPriority 0.05、tierAffinity 0.05），4 种模式包，bandit 探索，渐进式冷却。

10. **Docker：** Dockerfile 有两个目标：`runner-base` 和 `runner-cli`。`docker-compose.yml` 用于开发（3 个 profile），`docker-compose.prod.yml` 用于生产（端口 20130）。

11. **Electron 桌面应用** 位于 `electron/`，含 main.js 和 preload.js。通过 `npm run electron:build` 构建（支持 Windows、macOS、Linux）。

12. **定价数据** 从 LiteLLM 同步至 `src/lib/pricingSync.ts`。使用 `sync_pricing` MCP 工具或 API 端点。

13. **记忆系统** 位于 `src/lib/memory/`，提供提取、注入、检索、摘要和持久存储。通过 MCP 记忆工具和 `/api/memory/` API 对外暴露。

14. **技能系统** 位于 `src/lib/skills/`，提供注册表、执行器、沙箱隔离、内置技能、自定义技能支持、请求拦截和上下文注入。通过 MCP 技能工具和 `/api/skills/` API 对外暴露。

15. **所有校验使用 Zod v4**。从 `zod` 包导入。服务商 Schema 在模块加载时校验。

16. **Context Relay 策略**（`context-relay`）分两层实现：`combo.ts` 决定在成功轮次后是否生成交接；`chat.ts` 在账户解析后才注入交接信息。交接数据存储在 `context_handoffs` SQLite 表中。配置项：`handoffThreshold`、`handoffModel`、`handoffProviders`。

17. **代理强制全面覆盖**：Token 健康检查按连接解析代理，服务商校验在 `runWithProxyContext` 中包装，代理分发器使用 `undici.fetch()` 而非 Node 内置 `fetch()` 以避免 Node 22 上的分发器不兼容问题。

18. **Node.js 24+ 兼容性**：登录页面（`/api/settings/require-login`）检测 Node.js 版本，发送 `nodeVersion`/`nodeCompatible` 字段。当 `nodeCompatible` 为 false 时，登录 UI 渲染警告横幅。

19. **云代理** 位于 `src/lib/cloudAgent/` — 三个外部自主编程代理（Codex Cloud、Devin、Jules），任务生命周期端点位于 `/api/v1/agents/tasks/`。须使用管理认证，而非客户端认证。

20. **安全护栏框架** 位于 `src/lib/guardrails/` — 热重载注册表。内置（按优先级）：`vision-bridge`（5）→ `pii-masker`（10）→ `prompt-injection`（20）。Fail-open 模式：异常绝不阻塞流量。可通过 `x-omniroute-disabled-guardrails` 头进行逐请求关闭。

21. **授权管线**（`src/server/authz/`）：每个请求被分类为 `PUBLIC`、`CLIENT_API` 或 `MANAGEMENT`，然后经过策略 + 强制执行阶段。见 `docs/architecture/AUTHZ_GUIDE.md`。

22. **三种容灾层各司其职**（请勿混淆）：
    - **服务商熔断器**（`src/shared/utils/circuitBreaker.ts`）— 整个服务商范围。
    - **连接冷却**（`src/sse/services/auth.ts::markAccountUnavailable`）— 单个密钥/账户范围。
    - **模型锁定**（`open-sse/services/accountFallback.ts`）— 服务商 + 连接 + 模型范围。
