# SOTA — wasmtime vs wasmer for Plugin Runtime (side-13)

**Date:** 2026-06-20 | **Task:** side-13 | **Fleet: Rust-first**

## Background

**wasmtime** (Bytecode Alliance, Rust-native, Apache-2.0, ~50K LOC):

- Backend: Cranelift (fast compile, slightly slower steady-state than LLVM).
- Standards: WASI preview1 + preview2 (Component Model), `wasi-http`, `wasi-config`, threads, SIMD.
- Embed API: `wasmtime::Engine`, `Store`, `Linker`, `Instance`. First-class `component::bindgen!` macro for typed host bindings from `.wit` files.
- Current line: 27.x (June 2026), monthly cadence.

**wasmer** (Wasmer Inc., MIT runtime + custom compiler backends):

- Backends: Cranelift (default), LLVM (slow compile, fast runtime), Singlepass (AOT).
- Standards: WASI preview1 stable; preview2 in beta in 4.x. No Component Model `bindgen!` yet.
- License: MIT runtime, closed-source WASI-X extension tier.
- Current line: 4.x, ~6-week cadence.

## Verdict: **ADOPT wasmtime** (with a wasmer escape hatch)

The fleet has **no production plugin runtime** today; this is "if we add one, pick the right substrate."

## Why

1. **Standards trajectory.** The WASM Component Model is the future of polyglot plugin interfaces (canonicalized 2024, broad adoption by 2026). wasmtime supports it on stable; wasmer is catching up.
2. **Cranelift is the right default.** Plugin compilation happens once at install/load; runtime speed is bounded by per-call host-bindings overhead, not raw FP. Cranelift's compile-time edge is the bigger win.
3. **Bytecode Alliance governance.** wasmtime is governed by Fastly/Microsoft/Intel/Cosmonic — no single-vendor lock. Wasmer Inc. is one company with closed-source WASI-X.
4. **Embedded Rust ergonomics.** `wasmtime::component::bindgen!` generates typed host trait impls from `.wit` files — the same shape as `pheno-port-adapter` `Port` traits. Mental model maps 1:1 to fleet substrate style.
5. **Fleet relevance**: `phenotype-hub` (framework, dynamic lib loading today), `pheno-mcp-router` (provider adapters), `phenotype-tooling` (next-gen monorepo bootstrap), `pheno-otel` (otel-collector's own WASM processor uses wasmtime).

## Action items

1. **Spike**: `pheno-plugin-runtime` as a new substrate — wrap `wasmtime::Engine` + `wasmtime::component::bindgen!` behind a `Plugin: Port` trait so the rest of the fleet adopts incrementally. Crate budget: <3000 LOC. ~1 PR-week.
2. **First concrete consumer**: a `LlmWasmAdapter` — port one existing provider (e.g., the OpenAI-compat adapter from `pheno-mcp-router` PR #3) to a `.wasm` component. Pass criterion: round-trip latency within 2× of native impl.
3. **WIT interface authoring standard**: any new plugin contract authored as a `.wit` file in `pheno-plugin-runtime/wit/`. Free interop with JS (`jco`) and Python (`componentize-py`) plugin authors.
4. **Do not adopt wasmer as a co-runtime** — doubles the security review surface (different CVE feeds, supply chain). wasmer escape hatch is "if wasmtime misses a critical WASI capability," not "let's evaluate both at the call site."

## When wasmer is the right call (escape hatch)

- Cold-start <1ms is a hard requirement — wasmer's Singlepass backend has a slight edge; wasmtime 27.x has closed the gap to ~5%.
- Need for LLVM AOT (`.so` pre-compiled modules). wasmtime is moving toward AOT via `wasmtime compile` but not the same product shape yet.
- Embedded WASI-X extensions (closed-source) — not on our roadmap.

## What it is NOT a fit for

- **Hot-path in-process code** — WASM has ~10–50 ns/call overhead vs native FFI; if a "plugin" runs in a tight loop, do not make it a plugin.
- **GPU / SIMD-heavy compute** — WASM SIMD works but performance is uneven across hosts; offload to native.
- **Plugins that must call our internal Rust types directly** — WASM's value-prop is the boundary; if the boundary must be Rust-aware, dynamic loading (dlopen) is simpler.
