# Guardrail: binary size regression check for mobile builds (side-56)

**Date:** 2026-06-20
**Owner:** pheno-mobile circle
**Status:** Proposed (FLEET_DAG.db: side-56 DONE — file artifact pending)

## Current state

Mobile crates (`pheno-mobile-*`, `phenotype-mobile-*`) build `.ipa` and
`.aab` artifacts that ship to TestFlight / internal track. There is no
size budget and no CI signal when a PR adds 800KB to the release binary.
The 2026-Q2 mobile rollout hit the 200MB App Store cellular download
limit on iOS because of an unflagged `tokio` feature-flag flip that pulled
in 14MB of regex engines.

## Proposed guardrail

A CI matrix that builds the release binary for `aarch64-apple-ios` (and
`arm64-v8a` for Android), records the artifact size, and fails the check
if it exceeds a per-repo budget. Budgets are stored in
`.github/mobile-size-budget.json` and incremented only via PR with
explicit justification.

## Concrete implementation

`.github/workflows/mobile-size.yml` (12 LOC):

```yaml
name: mobile-size
on: [pull_request]
jobs:
  check:
    runs-on: macos-14
    steps:
      - uses: actions/checkout@v4
      - run: cargo build --release --target aarch64-apple-ios
      - run: |
          SIZE=$(stat -f%z target/aarch64-apple-ios/release/libpheno.a)
          BUDGET=$(jq -r '.aarch64_apple_ios' .github/mobile-size-budget.json)
          echo "size=$SIZE budget=$BUDGET"
          [ "$SIZE" -le "$BUDGET" ]
```

`.github/mobile-size-budget.json` (5 LOC):

```json
{
  "aarch64_apple_ios": 20971520,
  "arm64_v8a": 15728640
}
```
