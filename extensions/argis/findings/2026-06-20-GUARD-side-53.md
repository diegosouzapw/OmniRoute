# Guardrail: CI check for stale issue references (side-53)

**Date:** 2026-06-20
**Owner:** pheno-ci-templates circle
**Status:** Proposed (FLEET_DAG.db: side-53 DONE — file artifact pending)

## Current state

PR descriptions and commit messages routinely reference issues by number
(`#1234`) without verifying the issue is still open. Once an issue is
closed (resolved, won't-fix, duplicate), the reference becomes noise in
the timeline and confuses `gh issue list --search`. In 2026-Q2 we counted
47 merged PRs referencing already-closed issues — wasted review attention
on dead context.

## Proposed guardrail

A CI step on `pull_request` that extracts every `#NNN` reference from the
PR body and commit messages, queries each via `gh issue view`, and fails
the check if any referenced issue is in the `CLOSED` state. Skips
references inside fenced code blocks (legitimate doc/test examples).

## Concrete implementation

`.github/workflows/issue-refs.yml` (12 LOC):

```yaml
name: issue-refs
on: [pull_request]
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: |
          REFS=$(gh pr view "$PR_NUM" --json body -q .body \
            | grep -oE '#[0-9]+' | sort -u)
          for r in $REFS; do
            STATE=$(gh issue view "${r#\#}" --json state -q .state)
            if [ "$STATE" = "CLOSED" ]; then
              echo "::error::Reference $r is CLOSED"; exit 1
            fi
          done
        env:
          PR_NUM: ${{ github.event.pull_request.number }}
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

## Effort

**M** — 1-2h including edge cases (cross-repo references, transfer state,
private issues). No repo changes needed beyond the workflow.
