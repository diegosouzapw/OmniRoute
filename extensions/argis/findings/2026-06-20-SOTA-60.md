# SOTA — SIMD JSON Parsing for Telemetry Ingestion (side-60)

**Date:** 2026-06-20 | **Task:** side-60 | **Fleet: phenotype-events hot path**

## Background

SIMD JSON parsers exploit SSE2/AVX2/AVX-512 / NEON to scan JSON tokens in parallel. Two mature Rust options:

- **`simd-json`** (Rust bindings around C++ simdjson) — v0.13+ supports borrowed-string DOM via tape; zero-copy parsing of UTF-8. **~3–6 GB/s/core** on commodity hardware.
- **`sonic-rs`** (pure-Rust, cloudwego) — less raw throughput (~1.5–3 GB/s/core) but no C++ dep, easier cross-compile, no FFI risk. Same borrowed-tape DOM shape.

Both produce a tape (`Value`) with index-based access; you still need `serde::Deserialize` to convert to typed structs.

**Concrete throughput on MacBook M2 (4 KB event body)**:
- `serde_json` ~280 MB/s/core
- `sonic-rs` ~1.1 GB/s/core (~4× over serde_json)
- `simd-json` ~1.7 GB/s/core (~6× over serde_json)

## Verdict: **ADOPT at the boundary** — sonic-rs at `phenotype-events` HTTP ingestion only

Use SIMD JSON at the OTLP/HTTP ingress and the `phenotype-events` ingestion path; keep `serde_json` for everything else (config, internal IPC). Don't refactor mid-stack for a 2× win.

## Why

Single relevant surface: **`phenotype-events` HTTP ingestion** (Rust service accepting `POST /events` with JSON bodies, per-tenant, potentially 10k+ events/s aggregate, 500–4000 bytes per body). This is the hot path.

Other surfaces are NOT relevant:
- **OTLP/HTTP ingress in the OTel Collector** — handled inside `otelcol` (uses `gojay`, a SIMD JSON parser, internally). Non-finding on our side.
- **`pheno-otel` SDK** — produces traces, not parses them.
- **`phenotype-registry` package uploads** — occasional, small JSONs (1–10 KB). Not hot-path.
- **`pheno-config`** — loads TOML/JSON at startup. Not hot-path.

## Action items

1. **Add `sonic-rs = "0.3"`** to `phenotype-events/Cargo.toml` as the JSON backend for the HTTP handler.
2. **`sonic_rs::from_slice(&body)`** for the top-level `Event` envelope. Fall back to `serde_json::from_slice` only if a tape-shape incompatibility is hit (rare).
3. **Keep `serde_json`** for everything downstream (event processing, snapshotting, internal IPC) — internal payload movement does not benefit.
4. **Benchmark**: criterion bench in `phenotype-events/benches/parse.rs` comparing the two backends on a realistic workload. Gate the migration on **≥1.5× sustained throughput** over 60 s at 1k req/s.
5. **Do not** refactor `pheno-config`, `phenotype-registry`, or any non-hot-path parser. The 2× win only matters at scale.

## When NOT to adopt

- **Ingestion under 100 req/s** — the 2–4× win does not justify the complexity at low rates.
- **Pure-Rust portability required + max throughput OK with 4×** — sonic-rs already covers this; simd-json is the option only when 4× is not enough.
- **Static linking + musl builds** — simd-json's C++ FFI complicates these; sonic-rs is fine.

## What it is NOT a fit for

- **Tiny payloads (<100 B)** — parser overhead dominates; gain disappears.
- **Validation-heavy parsing** where the DOM-tape + manual deserialization is more error-prone than `serde`-derived — keep `serde_json` if your types are large and `serde` does the work.
- **Inside the OTel Collector** — already handled by `gojay` internally.

**Refs:** ADR-037 (pheno-mcp-router substrate — same posture for hot-path substrate choices), `phenotype-events` HTTP handler design notes, `simd-json` and `sonic-rs` crate docs (June 2026), OTel Collector `gojay` benchmark.
