# SOTA — PostgreSQL 16 JSONB Path Indexing for Event Store (side-36)

**Date:** 2026-06-20 | **Task:** side-36 | **Fleet: phenotype-events durable store**

## Background

PostgreSQL 15 added `JSON_TABLE` / `JSON_PATH_EXISTS` / `JSON_PATH_QUERY` (SQL/JSON path language, RFC 9535). PG16 tightened the implementation:

- Better optimizer integration.
- Smaller index sizes via `USING gin (jsonb_path_ops)` — ~3× smaller than the older `jsonb_ops` GIN.
- Predicate pushdown for path-filtered scans.

In practice: index a deep key like `(event->'payload'->>'tenant_id')` with a GIN path-ops index and serve point lookups + containment queries (`event @> '{"kind":"order.created"}'`) without a separate denormalized column.

**Concrete sizing reference**: `USING gin (payload jsonb_path_ops)` on a 100M-event table (avg payload 800 B) is ~6–9 GB. Read latency for `kind = 'order.created' AND payload->>'tenant_id' = 'acme' AND ts > now() - interval '1 hour'`: **<5 ms p99** on a 4-vCPU PG16 instance.

## Verdict: **ADOPT** (conditional on `phenotype-events` PG migration)

If `phenotype-events` stays on SQLite, defer. If migrating to PG, adopt path-ops GIN from day one.

## Why

Fleet relevance:

- `phenotype-events` — durable event store substrate. Today uses `sqlx` over SQLite (per ADR-035B). Migration target is "any PG-class RDBMS."
- `phenotype-registry` — already runs on SQLite local / PG fleet mode. JSONB path indexing lets us index `package.versions[*].yanked == false` predicates that today require a scan.
- `pheno-worklog-schema` v2.1 ingestion — secondary read path.

The relevant substrate is **`phenotype-events`**. SQLite has `json_extract` + `json1` extension but no equivalent of path-ops GIN; the win does NOT transfer.

## Action items

1. **Adopt now** if `phenotype-events` migrates SQLite → PostgreSQL as the canonical durable backend (SQLite stays as dev/test convenience). Path-ops GIN on `(payload, kind)` makes tenant-scoped replays and `kind = 'X' AND ts > T` lookups sub-millisecond.
2. **Migration**: `phenotype-events/migrations/0007_jsonb_path_ops.sql` — create GIN path-ops index per (kind, payload) tuple at schema-migration time.
3. **Default to `jsonb_path_ops`**; fall back to `jsonb_ops` only when wildcard key queries (`payload ? 'foo'`) are first-class.
4. **Open ADR-strawman** against ADR-035B pointing at this finding; pull into the substrate spec on next touch.

## When NOT to adopt

- **Fleet stays on SQLite** for the event store — no equivalent; defer.
- **Events already denormalized** to flat columns — don't pay the JSONB tax if normalized.
- **Cardinality too low** — if you have <1M events or query patterns are always by primary key, a BTREE on `(ts, kind)` is enough.

**Refs:** ADR-035B (event-bus substrate consolidation), ADR-012 (pheno-tracing canonical), `phenotype-events` schema migration history, PG16 release notes §JSON, RFC 9535 (JSONPath 1.0).
