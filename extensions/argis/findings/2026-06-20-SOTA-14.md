# SOTA — litellm vs Raw OpenAI for Coaching Provider (side-14)

**Date:** 2026-06-20 | **Task:** side-14 | **Fleet: Python (phenotype-journeys)**

## Background

**litellm** (Python, MIT, ~10K LOC user-facing):

- One `completion(model="gpt-4o", messages=...)` call routing to 100+ providers (OpenAI, Anthropic, Bedrock, Vertex, Azure, Ollama, llama-cpp, vLLM).
- Standardized response shape, retry/backoff, cost computation (cents/call), budget alerts.
- Drop-in OpenAI-compatible proxy: `litellm --model openai/gpt-4o --model anthropic/claude-sonnet-4`.
- Streaming, function-calling, JSON-mode, vision, audio — unified across providers.
- Current: 1.51.x (June 2026), weekly cadence.

**Raw OpenAI SDK** (`openai-python`):

- Direct call to one provider at a time. Excellent type safety, great streaming ergonomics.
- For multi-provider fan-out, we'd re-implement routing ourselves.

The fleet's only "coaching provider" surface today is `phenotype-journeys` (Python service, structured growth-journey flows). Currently a thin wrapper over OpenAI Python SDK with `gpt-4o` as the only model.

## Verdict: **ADOPT litellm** for `phenotype-journeys`

## Why

Three concrete pressures push toward multi-provider:

1. **Cost** — coaching calls are higher-token than chat (multi-turn + tool use). Per-call cost is dominant ops expense. litellm's per-model cost tracking + budget alerts enable automatic switch to cheaper model when monthly spend crosses threshold.
2. **Provider fall-back** — OpenAI had two regional outages in Q1-2026 that degraded `phenotype-journeys` SLA. Anthropic + llama-cpp fall-back would close this.
3. **Self-hosted option** — ADR-029 absorbed the llama-cpp devops setup; we now have a path to run open-weights locally. litellm supports llama-cpp as a first-class provider; raw OpenAI does not.

## Action items

1. **`phenotype-journeys` PR-1**: swap raw OpenAI SDK for litellm `completion()` calls. Keep OpenAI SDK as transitive dep; litellm already vendors it. Pass existing test suite.
2. **`phenotype-journeys` PR-2**: add Anthropic as configured fall-back. Assert fall-back path in an integration test.
3. **`phenotype-journeys/config/models.yaml`**:
   ```yaml
   default: openai/gpt-4o
   fallbacks:
     - anthropic/claude-sonnet-4-20250514
     - ollama/llama3.3-70b
   budget:
     monthly_cap_usd: 500
     per_call_max_usd: 0.50
   ```
4. **`phenotype-ops` PR-1**: add `litellm-proxy/` agent setup (Dockerfile + compose + README). Each journeys instance connects to `http://localhost:4000/v1` instead of `api.openai.com` directly. Unlocks log-and-replay, request budgeting, fleet-wide fall-back. <300 LOC.
5. **`phenotype-journeys` PR-3**: budget-alert integration with `pheno-otel`.

**Cost / complexity trade-off:**
- litellm dep cost: +15 MB Python wheel, ~3s longer cold-start. Trivial.
- Proxy sidecar cost: +1 process, +50 MB RAM, ~5 lines of compose. Trivial.
- Operational savings: estimated **40% reduction in per-call cost** via model fall-back (preliminary; needs 30-day pilot).
- Time-to-multi-provider: raw SDK ~3 PR-weeks/provider (auth, error mapping, retry); litellm ~1 PR-day total.

## When NOT to adopt

- **Latency-sensitive hot loops** — litellm adds ~5–10 ms/call routing overhead. For realtime audio / sub-200ms responses, raw SDK is faster.
- **Strict type-safety** — litellm responses are dict-shaped, not Pydantic. Wrap in Pydantic adapter if compile-time guarantees needed.
- **Single-provider apps** — if we commit to OpenAI-only for 18+ months, litellm routing is dead weight.
- **Advanced SDK features** (custom tools, assistant threads, file uploads, realtime audio) — litellm is a thin layer; use raw SDK for these. Use litellm as default for `completion()` / `embedding()`.
