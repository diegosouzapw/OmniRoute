# side-39: Dockerfile multi-stage builds audit

**Date:** 2026-06-20
**Scope:** `*.Dockerfile*` and `Dockerfile.*` across the meta-repo, excluding `.git/`, `target/`, `node_modules/`, `.venv/`.

## What I checked

Concrete grep commands run from the meta-repo root:

```bash
# 1. Total Dockerfiles in the meta-repo
find . -name "Dockerfile*" -not -path "*/.git/*" -not -path "*/target/*" \
     -not -path "*/node_modules/*" | wc -l
# → 213

# 2. FROM lines that are multi-stage (FROM <image> AS <stage>)
grep -rE "FROM.*AS" . --include="Dockerfile*" --include="*.dockerfile" \
     | grep -v "node_modules|/.git/" | wc -l
# → 109

# 3. Total FROM declarations
grep -rE "^FROM " . --include="Dockerfile*" --include="*.dockerfile" \
     | grep -v "node_modules|/.git/" | wc -l
# → 307

# 4. Single-stage Dockerfiles (no "AS" alias anywhere)
find . -maxdepth 3 -name "Dockerfile*" -not -path "*/.git/*" \
     -not -path "*/.venv/*" | xargs grep -L "AS "
# → ~20+ single-stage files (no build/runtime separation)
```

## Findings

**Strong multi-stage (P0 compliant):**
- `OmniRoute/Dockerfile:1-9` — 5 stages: `base`, `builder`, `runner-base`, `runner-web`, `runner-cli` (exemplary separation).
- `PhenoProject/rust/Planify/apps/proxy/Dockerfile.ce`, `apps/web/Dockerfile.web`, `apps/admin/Dockerfile.admin`, `apps/live/Dockerfile.live` — separate build/runtime stages.

**Single-stage (no `AS`, build tooling shipped to runtime image):**
- `helios-cli/Dockerfile` (line 1+: `FROM rust:1.83-slim` only — no `AS`).
- `phenotype-ops/review-surface/Dockerfile` (single stage, contains build deps).
- `PolicyStack/Dockerfile` (single stage).
- `pheno/Dockerfile.rust`, `pheno/agileplus/Dockerfile.rust`, `pheno/python/Dockerfile.python` (single stage).
- `HexaKit/Dockerfile.rust`, `HexaKit/python/Dockerfile.python`, `HexaKit/agileplus/Dockerfile.rust`, `HexaKit/agileplus/python/Dockerfile.python`, `HexaKit/agileplus-mcp/Dockerfile` (5 single-stage files).
- `PhenoDevOps/Dockerfile.rust`, `PhenoDevOps/python/Dockerfile.python`, `PhenoDevOps/agileplus/Dockerfile.rust`, `PhenoDevOps/agileplus/python/Dockerfile.python`, `PhenoDevOps/agileplus/agileplus-mcp/Dockerfile`, `PhenoDevOps/agileplus-mcp/Dockerfile` (6 single-stage).
- `AgilePlus/Dockerfile.rust`, `AgilePlus/python/Dockerfile.python`, `AgilePlus/agileplus-mcp/Dockerfile` (3 single-stage).
- `Dino/Dockerfile` (single stage; engine-heavy).
- `HeliosCLI/.devcontainer/Dockerfile` (single stage).
- `FocalPoint/AtomsBot-2nd/Dockerfile` (single stage).

Approximately **20+ of 213 Dockerfiles** have no multi-stage separation, meaning build toolchains (cargo, gcc, make) ship to the production image. Images like `pheno/Dockerfile.rust` retain the full Rust toolchain in the final layer, inflating image size and attack surface.

## Severity

**P1.** Multi-stage is the industry default (Docker docs, Snyk best-practice, CIS Docker 4.6). Most core/active repos (`OmniRoute`, `PhenoProject/Planify`) comply; a long tail of older/template/copy-paste Dockerfiles (especially in the `pheno/`, `HexaKit/`, `PhenoDevOps/`, `AgilePlus/` template families) does not. Not P0 because runtime services still build, but supply-chain + image-size hygiene is degraded.

## Recommended action

1. **Wave 1 (this sprint):** Convert the 6 `PhenoDevOps/.../Dockerfile*` and 5 `HexaKit/.../Dockerfile*` files to two-stage (builder + runtime) using `gcr.io/distroless/cc-debian12` or `debian:bookworm-slim` as runtime. These are actively scaffolded.
2. **Wave 2 (next sprint):** Same conversion for `pheno/Dockerfile.rust`, `pheno/python/Dockerfile.python`, `AgilePlus/Dockerfile.rust`, `Dino/Dockerfile`.
3. **Policy gate:** add a `policy-gate.yml` rule that fails any `Dockerfile` lacking `AS` keyword in a build stage (allowlist: `.devcontainer/Dockerfile`).
4. **Ref:** ADR-023 Rule 3.1 (substrate quality bar) does not yet cover container hygiene — propose ADR-040-style gate extension.
