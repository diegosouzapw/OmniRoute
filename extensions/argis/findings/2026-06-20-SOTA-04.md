# SOTA — tokio-console for Runtime Observability (side-04)

**Date:** 2026-06-20 | **Task:** side-04 | **Fleet: Tokio 1.40+, Rust 1.78+**

## Background

`tokio-console` (0.1.13+, June 2026) is the Tokio team's debugger for the async runtime. Two-line wire-up via `console-subscriber::init()` (dev) or layered behind `tracing-subscriber` (prod). Provides:

- **Task view** — every spawned task with current state (`Idle`/`Scheduled`/`Running`/`Yielded`), waker chain, polls, total busy time.
- **Resource view** — current + historical `Semaphore` permits, `Notify` waiters, mutex holders, `mpsc` depths.
- **Waker view** — diagnose missed wakeups or starvation.
- **Stream/Sink view** — pending messages on internal channels.

CLI: `tokio-console http://localhost:6669` (TCP/Unix domain socket; TLS opt-in via unstable `with_tls` in 0.1.x, stable in 0.2). Overhead <1% per Tokio team measurements; +200 KB binary size.

OTLP + `tracing-subscriber` already covers structured spans + export but does NOT give the live "what is every task doing right now" view — that is exactly tokio-console's gap-fill.

## Verdict: **ADOPT** in dev/CI, gated for prod

Wire `console-subscriber` into `pheno-tracing::init_tracing` as an opt-in layer. Default off in release builds; default on in `cfg(debug_assertions)` and `PHENO_TRACING_CONSOLE=1` env.

## Why

Fleet services that would benefit (all currently on Tokio):

- `pheno-events` — one task per consumer; hard to debug "why is consumer X not draining?" without per-task state.
- `phenotype-hub` — long-lived tasks for consumer registration, dispatch loop. A wedged consumer is invisible until logs trip.
- `phenotype-gateway` — per-connection task. Need "which connections are blocked on which downstream" during incidents.
- `pheno-mcp-router` — each `LlmPort::call` is a task. Currently time-out at consumer level; per-task state would show which provider is slow.

## Action items

1. **`pheno-tracing` PR**: add `console-subscriber` opt-in layer in `init_tracing`. Default off in release; on in dev + `PHENO_TRACING_CONSOLE=1`. ~50 LOC + 1 dep.
2. **CI workflow** (in integration-test job): `TOKIO_CONSOLE_BIND=127.0.0.1:6669`, run `tokio-console --no-palette --output json` for 30s, snapshot task count histogram, fail test if any task is `Running > 5s` without progress. Catches "one consumer wedged" bug class without human intervention.
3. **Dev wrapper**: `just trace-console` target that runs `cargo run --bin <service>` with console subscriber pre-wired.
4. **Reuse OTLP path**: `console-subscriber` exports via `tracing` spans, so existing `pheno-otel` ingest gets task-state events for free if the right `Layer` is hooked.
5. **Security**: do not expose port 6669 outside localhost without TLS. Default bind to `127.0.0.1`.

## What it is NOT a fit for

- **Headless CLI tools** that run and exit in <1s (no live state to inspect).
- **Production release builds** without explicit opt-in (debug surface + info-disclosure risk).
- **`wasm32` targets** — Tokio requires 64-bit atomics; non-issue for the fleet (all 64-bit native).
