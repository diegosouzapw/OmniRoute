# SOTA — NATS JetStream vs RabbitMQ for Event Bus (side-37)

**Date:** 2026-06-20 | **Task:** side-37 | **Fleet: phenotype-bus inter-service transport**

## Background

**RabbitMQ 3.13+** — classic AMQP broker. Quorum queues are now the default for HA (replaced mirrored queues). Streams (GA in 3.9) give append-only log semantics with replay. Operators manage via `rabbitmqctl` + management plugin.

**NATS JetStream** — persistence layer over NATS 2.x. File/streaming storage, subject-based addressing, ack/nak semantics, replay by sequence or timestamp. Native clustering; manage via `nats` CLI.

Both are mature. Consumer count today: ~5 services (gateway, registry, events store, otel collector, observability). Projected at 71-pillar maturity: 10–15. Either broker handles this fine.

## Verdict: **DEFER NATS** — Adopt RabbitMQ 3.13 (quorum queues + Streams) for `phenotype-bus`'s first inter-service transport

NATS stays in the back pocket; this finding deprioritizes, not closes out.

## Why

| Dimension | RabbitMQ 3.13 | NATS JetStream | Winner |
|---|---|---|---|
| Ops burden | Erlang VM, ~30 knobs, quorum config | Go binary, ~10 knobs, JetStream default-on | NATS |
| Replay | Streams (add-on) | First-class (sequence/timestamp) | NATS |
| Multi-tenancy | vhosts + per-vhost ACLs | accounts + scoped signing keys | NATS |
| Memory footprint (idle) | ~200 MB | ~30 MB | NATS |
| Throughput ceiling | ~50k msg/s single node | ~10–15M msg/s single node | NATS |
| HA story | quorum queues (Raft) + shovel/federation | native clustering (Raft-ish gossip) | tie |
| Mental model fit | queue-per-consumer-group, durable exchange | subjects, streams, consumers as separate objects | **RabbitMQ** |

**Decisive column for today**: both are greenfield, but RabbitMQ's mental model (queue-per-consumer-group, durable exchange) maps cleanly to the current `phenotype-bus::Topic` API. NATS's subject/stream/consumer model is more powerful but requires more upfront design work — and the gain (replay, multi-tenancy) is not load-bearing at our consumer count.

## Action items

1. **`phenotype-bus` exposes `BrokerPort` trait** per ADR-014/038. Initial `AmqpBrokerAdapter` impl uses `lapin` (Rust AMQP 0.9.1 client).
2. **Streams added only when** a concrete consumer (likely `phenotype-events`) needs replay.
3. **Re-evaluation trigger**: consumer count crosses 50, or we need at-least-once + replay for `phenotype-events` sinks.

## When NATS JetStream is the right call (revisit triggers)

- Consumer count crosses 50 — NATS's higher throughput ceiling becomes load-bearing.
- Multi-tenancy requirement grows (per-tenant auth + quota) — NATS accounts are first-class.
- We want to fold messaging into a K8s-native deployment where NATS's smaller ops footprint matters.
- We need at-least-once + replay semantics as a baseline, not as an add-on (Streams on RabbitMQ work but require extra config).

## What it is NOT a fit for (RabbitMQ side)

- **Single-message multi-recipient fan-out** without per-consumer queues — both brokers handle this; RabbitMQ exchanges map directly.
- **Edge / IoT deployments** with intermittent connectivity — NATS leaf nodes are stronger here, but not a fleet need today.
