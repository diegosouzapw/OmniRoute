# side-44: Hardcoded timeouts vs context deadlines

**Date:** 2026-06-20
**Scope:** Timeout/duration patterns in Rust (`tokio::time::timeout`, `Duration::from_secs`) and Go (`context.WithTimeout`, `time.Second` literals).

## What I checked

```bash
# 1. Hardcoded duration literals in Rust
grep -rE "Duration::from_secs|Duration::from_millis|timeout\s*=\s*[0-9]" . \
     --include="*.rs" | grep -v "node_modules|/target/|/.git/" | head -50
# Findings: many in .trash-Civis (ignored), helios-cli, AuthKit, etc.

# 2. Context-aware timeouts in Rust
grep -rE "tokio::time::timeout|with_timeout" . --include="*.rs" \
     | grep -v "node_modules|/target/|/.git/" | head -20
# Findings: KWatch, helios-cli use it well.

# 3. Hardcoded time.Second literals in Go
grep -rE "[0-9]+\*time\.Second|time\.(Second|Minute|Millisecond)" . \
     --include="*.go" | grep -v "node_modules|/.git/" | head -20
```

## Findings

**Compliant — context propagation:**
- `KWatch/cmd/daemon.go` — `ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)` — derives context from background; OK for daemon startup.
- `KWatch/runner/runner.go` — `cmdCtx, cancel := context.WithTimeout(ctx, timeout)` — propagates parent `ctx` (good).
- `KWatch/mcp/server.go` — `context.WithTimeout(context.Background(), 12*time.Second)` and `18*time.Second` — hardcoded numbers but derived from `context.Background()`. Acceptable for the MCP subprocess bootstrap, but the literal values are not config-driven.
- `KWatch/server/server.go` — `shutdownCtx, cancel := context.WithTimeout(context.Background(), 5*time.Second)` for graceful shutdown. Standard pattern.
- `helios-cli/crates/harness_runner/src/lib.rs` — `match tokio::time::timeout(Duration::from_secs(timeout), cmd.output()).await` where `timeout` is a `secs: u64` method parameter (configurable). Good.
- `helios-cli/crates/harness_verify/src/runners.rs` — `tokio::time::timeout(...)` with caller-supplied durations. Good.
- `helios-cli/go/harness/cache.go` — `return context.WithTimeout(context.Background(), timeout)` with `timeout` parameter. Good.

**Non-compliant — hardcoded literals with no context propagation:**
- `helios-cli/crates/harness_orchestrator/src/root_manager.rs` — `tokio::time::sleep(tokio::time::Duration::from_millis(100)).await` hardcoded 100ms in a polling loop. Not a network call but a busy-wait pattern that should use `tokio::sync::Notify` or a configurable interval.
- `Configra/crates/settly/src/application/submission_tests.rs` — `tokio::time::sleep(Duration::from_millis(5)).await` (test code; acceptable).
- `KWatch/mcp/server.go` — `12*time.Second` and `18*time.Second` for MCP bootstrap. The two different literals in the same file suggest copy-paste rather than config-driven design. (P2.)
- `.trash-Civis-wt-l5-114-1781947944/crates/server/tests/ws_smoke.rs` — many `timeout(Duration::from_secs(2), ...)` and `timeout(Duration::from_secs(3), ...)` in tests. Hardcoded, but tests are typically exempt from config-driven timeouts. Not in scope (trash).
- `BytePort/frontend/web/src-tauri/vendor/aws-runtime/src/content_encoding.rs` — `tokio::time::timeout(timeout_duration, test_fut)` — uses a parameter; compliant.

**No context cancellation in some sleep calls:**
- `helios-cli/crates/harness_orchestrator/src/root_manager.rs` — `tokio::time::sleep(...).await` does not respect a select-on-cancel pattern. If the harness orchestrator is shut down mid-loop, the 100ms sleep still completes before the cancellation is observed. Minor (P3) — this is a polling loop, not a network call.

## Severity

**P2** (informational/hardening). Most of the fleet uses context-aware timeouts correctly (KWatch, helios-cli). The `KWatch/mcp/server.go:12*time.Second` / `18*time.Second` literal duplication is a code-smell but not a correctness bug. The `helios-cli` `root_manager.rs` 100ms hardcoded polling interval is the only pattern worth fixing (P3 polish).

This is **not P0 or P1** because the production request paths in `KWatch/runner/runner.go` and `helios-cli` correctly propagate `ctx`. The hardcoded literals are a config-extraction refactor (ADR-022 territory), not a correctness bug.

## Recommended action

1. **P3 this sprint:** `helios-cli/crates/harness_orchestrator/src/root_manager.rs` — replace hardcoded `100ms` with a configurable `poll_interval: Duration` field on the orchestrator config, defaulting to 100ms.
2. **P3 next sprint:** `KWatch/mcp/server.go` — extract the `12*time.Second` and `18*time.Second` literals into named constants (`mcpHandshakeTimeout`, `mcpAuthTimeout`) and consider promoting to config.
3. **P2 doc:** add `docs/timeout-policy.md` codifying: (a) all network ops take a `ctx` first arg, (b) all timeouts derive from `ctx` via `WithTimeout`, (c) hardcoded literals only in tests.
4. **Policy gate (P3):** add a clippy/lint rule via `cargo-deny` or `dylint` to flag `Duration::from_secs(<literal>)` in `src/` (not `tests/`) without a comment explaining why.
5. **Reference:** `KWatch/runner/runner.go:cmdCtx, cancel := context.WithTimeout(ctx, timeout)` is the canonical pattern; document it in the policy doc.
