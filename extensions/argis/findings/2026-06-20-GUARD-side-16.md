# Guardrail: property-based tests for wallet invariants (side-16)

**Date:** 2026-06-20
**Owner:** pheno-wallet circle
**Status:** Proposed (FLEET_DAG.db: side-16 DONE — file artifact pending)

## Current state

The wallet crate has ~25 unit tests using fixed-value deposit / withdraw
examples. Conservation invariants (`balance == sum(history))` and
non-negativity (`balance >= 0`) are **assumed**, not **asserted**. A
regression that double-counts a fee would not be caught by any existing
test until someone reads the diff.

## Proposed guardrail

Add a `proptest!` module that generates random op-sequences (deposits,
withdraws, transfers) and asserts both invariants after every step. Run as
part of the standard `cargo test` matrix — no nightly infra needed.

## Concrete implementation

`tests/wallet_invariants.rs` (14 LOC):

```rust
use proptest::prelude::*;
use pheno_wallet::{Wallet, Op, AssetId};

fn op() -> impl Strategy<Value = Op> {
    prop_oneof![
        (0u128..1_000_000).prop_map(|a| Op::Deposit { asset: AssetId::USDC, amount: a }),
        (0u128..500_000).prop_map(|a| Op::Withdraw { asset: AssetId::USDC, amount: a }),
    ]
}

proptest! {
    #[test]
    fn conservation_holds(ops in proptest::collection::vec(op(), 0..200)) {
        let mut w = Wallet::new();
        for o in ops { w.apply(o).unwrap(); }
        prop_assert_eq!(w.balance(AssetId::USDC), w.history_sum(AssetId::USDC));
        prop_assert!(w.balance(AssetId::USDC) >= 0);
    }
}
```

## Effort

**M** — depends on `Wallet::apply` returning `Result` (small refactor if
not), 1-2h to wire `proptest` dev-dep and write the strategy. No infra cost.
