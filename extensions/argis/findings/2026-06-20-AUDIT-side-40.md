# side-40: GitHub Actions pinned action versions

**Date:** 2026-06-20
**Scope:** `**/.github/workflows/*.yml` and `**/.airlock/workflows/*.yml` across the meta-repo.

## What I checked

```bash
# 1. SHA-pinned (most secure, supply-chain resistant)
grep -rE "uses:\s+[a-zA-Z0-9_.-]+/[a-zA-Z0-9_.-]+@[a-f0-9]{40}" . \
     --include="*.yml" --include="*.yaml" | grep -v "node_modules|/.git/"
# Example: .github/workflows/release.yml: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683

# 2. Version-pinned (tag, e.g. @v4) — supply chain risk on tag-rebinds
grep -rEh "uses:\s+\S+@v[0-9]" . --include="*.yml" --include="*.yaml" \
     | grep -v "node_modules|/.git/" | wc -l
# → 152 occurrences in core substrate repos (PhenoCompose, AuthKit, Configra,
#   phenotype-hub, phenotype-bus, phenotype-gateway, phenotype-registry)

# 3. Branch-pinned (mutable, full supply-chain exposure)
grep -rEh "uses:\s+\S+@(main|master)" . --include="*.yml" --include="*.yaml" \
     | grep -v "node_modules|airlock" | head -20
# → 13 occurrences in core repos; 22 across the meta-repo including
#   coverallsapp/github-action@master, ljharb/require-allow-edits@main,
#   trufflehog/actions/setup@main, KooshaPari/phenoShared/.github/workflows/codeql.yml@main
```

## Findings

**SHA-pinned (compliant):** Only a few workflows:
- `.github/workflows/release.yml:actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683`
- `.github/workflows/release.yml:actions/setup-go@f111f3307d8850f501ac008e886eec1fd1932a34`
- `.github/workflows/deny.yml:actions/checkout@de0fac2e4500dabe0009e67214ff5f5447ce83dd # v4.2.2`
- `.github/workflows/deny.yml:dtolnay/rust-toolchain@3c5f7ea28cd621ae0bf5283f0e981fb97b8a7af9 # stable`
- `.github/workflows/governance-drift.yml:actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683`

**Version-pinned (acceptable but rebindable):** 152 occurrences in 7 core substrate repos — the dominant pattern. Examples: `actions/checkout@v4`, `actions/setup-python@v5`, `astral-sh/setup-uv@v3`, `codecov/codecov-action@v4`.

**Branch-pinned (non-compliant, mutable):** 13+ occurrences in core repos. Examples:
- `trufflehog/actions/setup@main` (multiple workflow files)
- `KooshaPari/phenoShared/.github/workflows/codeql.yml@main`
- `KooshaPari/phenotype-infrakit/.github/workflows/coverage.yml@main`
- `coverallsapp/github-action@master`
- `ljharb/require-allow-edits@main`
- `braintree/security-workflows/.github/workflows/{codeql,dependency-review}.yml@main`

A workflow that `uses: third-party/repo@main` runs whatever HEAD is on the third-party repo's main branch at the time the workflow runs. A malicious force-push or compromised maintainer token can inject arbitrary code into our CI.

## Severity

**P1.** SHA-pinning is the GitHub-recommended practice (github.com/security/ hardened-workflows guide) and is the only defense against tag-rebinding and branch-push attacks. The fleet's 152 `@v3`/`@v4`/`@v5` pins are **acceptable** (tag impersonation requires explicit collaboration by the upstream maintainer + GitHub), but the 13 `@main`/`@master` pins are a **P1 supply-chain risk** because they execute whatever HEAD is. None are P0 because no active compromise has been observed; this is a hardening finding.

## Recommended action

1. **Immediate (this week):** Convert all `@main` / `@master` action references to SHA-pinned (e.g., `trufflesecurity/trufflehog@<sha> # v0.107.0`) in 13 known files. Add a `# <semver>` comment for human readability.
2. **Policy gate:** add a `policy-gate.yml` step using `step-security/harden-runner` to detect unpinned actions and fail the PR.
3. **Migration plan:** for the 152 `@v<n>` references, file a v12 track to convert to SHA-pinned (low risk; tag-pinning already accepted, but a 1-week deprecation cycle on each `@vN` is fleet norm per ADR-040).
4. **Dependabot config:** add `.github/dependabot.yml` with `package-ecosystem: github-actions` to auto-bump SHAs on upstream release.
5. **Exempt list (document):** `.airlock/workflows/main.yml` 7 lines using `airlock-hq/airlock/defaults/*@main` are **exempt** (airlock is the platform that owns the meta-orchestration; this is the trusted bootstrap).
