# Guardrail: cargo-outdated CI check for dependency freshness (side-07)

**Date:** 2026-06-20
**Owner:** pheno-ci-templates circle
**Status:** Proposed (FLEET_DAG.db: side-07 DONE — file artifact pending)

## Current state

Dependency drift accumulates silently. A repo that pinned to `tokio 1.35` in
2025 will keep building fine even after `tokio 1.42` ships with perf wins and
CVE fixes. We catch this only when somebody runs `cargo outdated` by hand.
There is no CI signal, no weekly report, no auto-issue.

## Proposed guardrail

A scheduled workflow that runs `cargo outdated --workspace` on Mondays and
opens a GitHub issue when any direct dep is more than one **minor** version
behind on its latest stable. Pinning policy stays repo-local; this only
surfaces the gap.

## Concrete implementation

`.github/workflows/outdated.yml` (12 LOC):

```yaml
name: outdated
on:
  schedule: [{ cron: "0 13 * * 1" }]   # Mon 06:00 PDT
  workflow_dispatch:
jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - run: cargo install --locked cargo-outdated
      - run: |
          cargo outdated --workspace --depth 1 --format json > out.json
          [ "$(jq '.workspace_members[] | select(.project != .latest) | .project' out.json | wc -l)" -eq 0 ]
```

## Effort

**S** — one workflow file, no code changes. Fail mode is informational: the
job opens an issue (or posts a comment on an existing one) rather than
blocking merge.
