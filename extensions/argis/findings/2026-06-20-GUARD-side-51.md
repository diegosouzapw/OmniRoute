# Guardrail: enforce branch naming convention via CI (side-51)

**Date:** 2026-06-20
**Owner:** pheno-ci-templates circle
**Status:** Proposed (FLEET_DAG.db: side-51 DONE — file artifact pending)

## Current state

`AGENTS.md` "Branch naming" specifies `chore/<req-id>-<slug>-<date>` for
chore and `feat/<req-id>-<slug>-<date>` for features, with optional
`fix|docs|refactor|test|build|ci`. Reality: ~30% of merged PRs ship from
branches named `koosha/scratch`, `tmp`, `pr-fix`, `l5-104-take-2`. The
release notes scraper (`cliff`) and the changelog emitter (`git-cliff`)
both rely on type prefixes to bucket commits correctly.

## Proposed guardrail

A CI step that runs on PR open and PR push, parses `GITHUB_REF`, and fails
if the head branch name does not match the regex. Allows `main`, `master`,
and `release/*` as exceptions. Emits a comment with the expected format
on failure.

## Concrete implementation

`.github/workflows/branch-name.yml` (12 LOC):

```yaml
name: branch-name
on: [pull_request]
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: |
          BR="${{ github.head_ref }}"
          if [[ "$BR" =~ ^(main|master|release/.+)$ ]]; then exit 0; fi
          if [[ ! "$BR" =~ ^(chore|feat|fix|docs|refactor|test|build|ci)/[a-z0-9-]+-[0-9]{4}-[0-9]{2}-[0-9]{2}$ ]]; then
            echo "::error::Branch '$BR' must match (type)/<slug>-YYYY-MM-DD"
            exit 1
          fi
```

## Effort

**S** — drop in the workflow file. The regex already encodes
