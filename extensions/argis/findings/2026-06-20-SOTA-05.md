# SOTA — rust-analyzer Performance for Large Workspaces (side-05)

**Date:** 2026-06-20 | **Task:** side-05 | **Fleet: 17+ crate workspace, Rust 1.78+**

## Background

Phenotype Rust fleet size over time:

- **2025-Q4:** ~9 member crates in `pheno` workspace.
- **2026-Q1:** 14 (added `pheno-predict`, `pheno-framework-lint`, `pheno-drift-detector`).
- **2026-Q2 (now):** 17+ (`pheno-capacity` extracted per ADR-036, `pheno-secret-scan` in flight).

rust-analyzer's incremental rebuild handles this well **per-crate**, but at the meta-repo level (where most editor work happens — single checkout with all crates visible) regresses linearly with crate count and quadratically with macro density. The fleet uses `tracing-subscriber` and `serde` derives heavily; both are macro-amplified.

**Symptom**: in a 17-crate workspace with `checkOnSave: true`, a single-character edit in `pheno-port-adapter/src/ports/mod.rs` triggers **~14s** of background work before IDE squiggles refresh. With the recommended settings below: **~2s** (7× improvement).

## Verdict: **ADOPT** workspace settings + proc-macro hygiene

~20 lines of `.vscode/settings.json` + a `Cargo.toml` metadata block. Zero new deps, no CI changes.

## Why

The fleet-wide host memory budget on the MacBook (orchestrator device) is 16 GB. The default `rust-analyzer` config hits O(N²) in the `pheno-events` → `pheno-bus` → `pheno-port-adapter` → `pheno-tracing` dependency chain. Mitigations are well-documented; the gain is real and reproducible.

## Action items

1. **`pheno/.vscode/settings.json`** — workspace-recommended settings:
   ```json
   {
     "rust-analyzer.cargo.buildScripts.enable": true,
     "rust-analyzer.cargo.allTargets": false,
     "rust-analyzer.cargo.features": "no_default_features",
     "rust-analyzer.workspace.symbol.search.scope": "workspace",
     "rust-analyzer.files.watcher": "server",
     "rust-analyzer.cargo.autoreload": true,
     "rust-analyzer.updates.channel": "stable",
     "rust-analyzer.diagnostics.debounce": 500,
     "rust-analyzer.diagnostics.experimental.enable": false
   }
   ```
   - `files.watcher: server` saves ~30% CPU on macOS steady-state (uses internal notify vs OS watcher roundtrip).
   - `allTargets: false` saves 20–40% initial load (skips examples/benchmarks metadata).
   - `features: no_default_features` + per-crate override for `pheno-tracing/otel` — avoids worst-case feature-set union.

2. **`pheno/Cargo.toml`** — workspace metadata block:
   ```toml
   [workspace.metadata.rust-analyzer]
   rust-analyzer.proc-macro.ignored = []
   ```
   Pin `serde_derive` + `thiserror` to one version in `Cargo.lock` to avoid proc-macro server deadlocks. Document in `pheno-cargo-template/README.md`.

3. **`pheno/AGENTS.md`** — 5-line "IDE Performance" paragraph pointing contributors at the settings file + the 16 GB MacBook budget. Quote the 14s → 2s symptom numbers so contributors can self-diagnose.

4. **Estimate**: ~30 min of work, no new deps, no CI changes. Land as docs-only chore in v11 tier-2 docs pass.

## What it is NOT a fit for

- **Single-crate workspaces** — none of these settings help; some (notably `files.watcher: server`) can hurt.
- **CI** — uses `cargo check` + `cargo clippy --workspace`, not the LSP. Don't try to "speed up CI with rust-analyzer settings."
- **Switching off rust-analyzer entirely** — `rustc` direct is slower on the dev loop. rust-analyzer is the right tool; the question is just config.

**Refs:** `pheno/.vscode/`, `pheno-cargo-template/README.md`, rust-analyzer release notes (2024-08+), ADR-038 (hexagonal port-adapter L4 policy — explains the macro-heavy pattern).
