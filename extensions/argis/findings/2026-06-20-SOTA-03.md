# SOTA тАФ `async-trait` тЖТ Native `async fn` in Trait (side-03)

**Date:** 2026-06-20 | **Task:** side-03 | **Fleet: Rust 1.78+**

## Background

Rust 1.75 (Dec 2023) stabilized `async fn` in trait (RFC 3185), removing the `Box::pin` + `Send`-bounds machinery that `async-trait` injected. State as of Rust 1.79 (June 2026):

- **Stable**: native `async fn` in non-`dyn` traits and `impl Trait` return positions.
- **Partial**: auto-`Send` for return-position `impl Future` is usable in 1.78+ for common cases; `trait_variant::make` or explicit `Box<dyn Future + Send>` still needed for `Send`-bound edge cases.
- **NOT stable**: `dyn Trait` with `async fn`. Tracking issue `rust-lang/rust#110937`. Expected 1.81+ (late 2026).
- **`async-trait` crate**: still maintained (v0.1.86, June 2026); the workaround for `dyn` cases for at least another 6тАУ12 months.

`async-trait` overhead is ~5тАУ10 ns per call тАФ not a bottleneck in our workload.

## Verdict: **DEFER** (partial adoption safe now)

Roughly **75%** of fleet `async-trait` usage (24 import sites across `pheno-port-adapter`, `pheno-mcp-router`, `pheno-tracing`, `pheno-events`) is locked behind `dyn Port` ergonomics. Only **25%** is migratable today.

## Why

| Crate | `async-trait`? | `dyn`-shaped? | Migrate now? |
|---|---|---|---|
| `pheno-port-adapter` | yes (11 sites) | yes (`dyn Port`) | **no** until `dyn async fn` lands |
| `pheno-mcp-router` | yes (7 sites) | yes (`Box<dyn LlmPort>`) | **no** |
| `pheno-events` | yes (2 sites) | no (static dispatch via generic enum) | **YES** |
| `pheno-tracing` | yes (2 sites) | yes (`Box<dyn Layer<S>>`) | **no** unless factory returns `impl Trait` |
| `pheno-bus` | no | n/a (typed channels) | already done |
| `pheno-config` | no | n/a (enum-based provider) | already done |

Bulk-converting now would force `pheno-port-adapter` and `pheno-mcp-router` back into `BoxFuture` land тАФ net-zero win and increased churn.

## Action items

1. **Now**: migrate `pheno-events::EventSink` to native `async fn` in trait (1 PR, ~30 LOC change, static dispatch тАФ zero risk). Land in v11 tier-2.
2. **Now**: add a one-paragraph note to `pheno-port-adapter/README.md` тАФ "new ports prefer native `async fn` where `dyn`-dispatch is not required; reserve `async-trait` for `dyn Port` shapes."
3. **Q4-2026 trigger**: when Rust 1.81 ships with stable `dyn async fn`, open a fleet-wide sweep тАФ drop `async-trait` dep from each crate, regenerate docs. Re-evaluate the 24 import sites.
4. **Any new substrate**: write ports native from day 1; reserve `async-trait` only for `dyn`-shaped additions.

## What it is NOT a fit for

- **Crates that genuinely need `dyn Port`** тАФ the port-adapter pattern's whole point is runtime swapping. Use `async-trait` until `dyn async fn` lands.
- **Performance-critical hot loops** тАФ `async-trait` overhead is not our bottleneck; do not chase nanoseconds.
- **Stable < 1.75 toolchains** тАФ non-issue for the fleet (1.78) but check before adopting in third-party forks.
