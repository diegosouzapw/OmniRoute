# SOTA — OpenTelemetry Collector vs Native Exporters (side-38)

**Date:** 2026-06-20 | **Task:** side-38 | **Fleet: 5 OTLP-producing services**

## Background

**OTel Collector** (`otelcol`) — vendor-neutral pipeline binary. Receivers (otlp, jaeger, zipkin, prometheus) → processors (batch, tail_sampling, attributes, filter, routing) → exporters (otlp, prometheus, logging, file, debug). Runs as sidecar/daemonset/agent. GA since OTel 1.0 (2022); current line **OTel Collector v0.108+** (June 2026).

**Native exporters** — every OTel SDK ships with direct exporters to vendors (Datadog, Honeycomb, Tempo, Jaeger, GCP, AWS X-Ray, etc.). You configure the SDK with the vendor endpoint + auth and skip the Collector.

OTel spec explicitly allows direct export — the Collector is not required. But it gives one place to do sampling, retries, attribute PII stripping, fan-out, and protocol translation.

## Verdict: **ADOPT the Collector**

Skip-the-Collector path is the legitimate "Hello World" path but does not fit any prod service in the fleet.

## Why

Five OTLP-producing services today: `phenotype-gateway`, `phenotype-registry`, `phenoMCP`, `phenoObservability`, `phenoEvents`. Five producers → one collector → one backend is the canonical topology. Skipping the collector means every service has to know the backend endpoint, retry policy, sampling config, and PII-strip rules — duplicated five times.

We hit **all four** triggers for the Collector today:
1. ≥2 services emit telemetry (we have 5).
2. Want sampling (head-based parent-based ratio = 0.1 in prod; moving to tail-based once SLOs are defined).
3. Want to fan out to >1 backend (primary OTLP + secondary file ring-buffer).
4. PII stripping is a concern (`*.password`, `*.token`, `http.request.header.authorization`).

## Action items

1. **`pheno-otel` default**: OTLP/gRPC to `http://localhost:4317` — Collector sidecar in dev, Collector deployment in prod. The Collector config (`otelcol.yaml`) lives in `phenotype-ops/agent-devops-setups/otel-collector/` per ADR-023 federated-service rules.
2. **Processors**: `batch` (10s window, 8192 batch size), `memory_limiter` (hard cap 512 MB), `tail_sampling` (per-service policies once >3 services), `attributes/redaction` for `*.password`, `*.token`, `http.request.header.authorization`.
3. **Exporters**: primary `otlp` to chosen backend; secondary `file` for dev-mode ring-buffer replay; `debug` only at `--log-level=debug`.
4. **Sampling default**: head-based parent-based ratio = 1.0 in dev, 0.1 in prod. Move to tail-based once error-budget SLOs are defined.
5. **Auth**: mTLS between service-sidecars and the Collector (ADR-046 federation mTLS scope); bearer-token auth to the backend.
6. **Health/Readiness**: Collector exposes `:13133` health and `:8888` Prometheus self-metrics — already wired in the deployment.

## When to skip the Collector (native exporters)

- **Single-service deployment**, single backend, no sampling, no PII concerns.
- **Greenfield dev spike** that won't outlive the day — direct OTLP to `http://localhost:4318` (Collector debug receiver) is fine.

Neither case applies to any prod service in the fleet today.

## What it is NOT a fit for (Collector side)

- **Ultra-low-latency hot paths** — Collector adds ~5–10 ms hop. If a service has a <1 ms p99 SLO and telemetry adds meaningful budget, sample aggressively or skip the hop.
- **Compliance-mandated direct-to-vendor** — some regulated environments require no intermediate. Not a fleet concern today.
