# Guardrail: PR size limit check (max 500 lines) (side-52)

**Date:** 2026-06-20
**Owner:** pheno-ci-templates circle
**Status:** Proposed (FLEET_DAG.db: side-52 DONE — file artifact pending)

## Current state

PRs in the fleet routinely exceed 800 LOC; the largest this quarter was
2,400 LOC (a one-shot absorption of `dispatch-mcp` content). The
`code-review-change-size` skill flags >800 lines as a soft warning, but
nothing in CI enforces it. Reviewers skim, bugs slip through, and the
post-mortem in `findings/2026-06-18-track8-self-merge-postmortem.md`
attributes a P1 incident partly to a 1,800-LOC PR that merged with two
reviewer approvals and zero deep reads.

## Proposed guardrail

A CI step on `pull_request` that computes the diff line count against the
base branch and applies a **two-tier gate**: hard fail at 500, hard warn
(label `size/xl`) at 300. Override requires an explicit `## size-override`
line in the PR body — bypass is logged and reviewable.

## Concrete implementation

`.github/workflows/pr-size.yml` (10 LOC):

```yaml
name: pr-size
on: [pull_request]
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }
      - run: |
          LINES=$(git diff --shortstat origin/${{ github.base_ref }}...HEAD \
            | awk '{print $4+$6}')
          if [ "$LINES" -gt 500 ]; then
            echo "::error::PR is $LINES lines (limit 500)"
            exit 1
          fi
          echo "PR size: $LINES lines"
```

## Effort

**S** — single workflow file. Tighter than `code-review-change-size`'s 800
guidance by design: PRs that need to exceed 500 should be **split**, not
exempted.
